  1. Introduction

    This SRS (Software Requirement Specification) is about Document Tagging System. Document Tagging System is a software which help one or a group of people to tag one or multiple documents. Since there are many Document Tagging System exist in the software market but none of them give as convenient way to use as we give. This software can be used in web and as a mobile app both in android and ios and Moddle so all the ways are given to the user. So the user can use this product anywhere he belongs. This SRS describe the Product purpose, product feature, function and non-functional requirements which guide anyone about what the product is all about.

    1. Purpose

      The name of our product is Document Tagging System which give a general idea what the product is all about. The purpose of this product is to give a convenient way to the user to tag his/her documents. In this system, a user can tag a whole document or he/she can tag a particular word or number of words. So all kinds of flexibility is given to the system. This system can be used in anywhere as this service is in the web and there also exists a friendly app version of this system in android and ios. The service can be integrated in system like Moodle. There are many options to share the document with other users or group.

      The intended audience of this system is developers, project managers, users, testers and documentation writers. As the document is written in a standard way which is IEEE standard so any one can read this document with no difficulties. But it is good to read the document as it is written rather than read here and there with on sequences.

    4. Product Scope

      As this system give the user all kinds of flexibility to use, the product scope is very good in the Document Tagging system. The goal of this product is to get all kind of users. To achieve the goal, this system give many friendly options to a user to tag a document. Here a user and group of user can give constraint to the document how it is tagged by a particular user. For this reason there come different tagging system like private, public, restricted, sequential restricted. Those options give the users of organization the flexibility how a document can be tagged.

  2. Overall Description
    1. Product Perspective

      Our Document Tagging System (DTS) is a self-contained product. User can use this product from web or from their personal mobile phone. This product can also be used as a third party. Using this product user can get all types of tagging facilities. From Moodle or other External System, User can use this product using Plug-In. We also provide client of client (COC) facility. In this product we interconnect many sub-system and build a large system. In this product we emphasis on Identity management for authentication of proper authorized user. We also provide Single Sign On (SSO) and OAuth for identification purposes.

    2. Product Function

      1. User can Tag Document

        2. User can Tag any part of the document.

      2. Edit Tag

        3. One user can Edit his/her Tag on the Document if he/she have privilage.

      3. Delete Tag

        4. One user can Delete his/her Tag on the Document if he/she have privilage.

      4. Share Tag

        5. One user can share his/her Tag on Document with Other user.

      5. Support two types of users :

        1. Single User
          2. For example: Anyone.
        2. Group User
          3. For example: Any Group, Company, Organization, Bank etc.

      6. All user can get two types of facility. These are:

        1. Subscribe
        2. Publish

      7. Add 3 types of Privileges for users these are:

        1. Private Tag
          2. Only the creator of the document can tag a private document.
        2. Public Tag
          3. Anyone of the compnay can Tag on Document without maintaining order.
        3. Sequential Tag
          4. Only permitted user can Tag on Document with Maintaining a strict Order.
        4. Restricted Tag
          5. Only Restricted or Permitted user can Tag on Document.

    3. User Classes and Characteristics

      The users of our Product mainly classified into two classes. They are Single User, Group User. Any individual user are single user. These user can individually use our product for tagging purpose. On the other hand, any group of users are Group users. These user can also use our product for Tagging purpose. Example of Group users are: Office Employee, Bank Employee, Students etc. Using this product Group user can tag the Document and share the tagging document with other users. In this case Group user can add privileges on the document. These privileges are private, Public, Sequential, Restricted.

    4. Operating Environments


      Our Product's web service can be used from any platform using through Google Chrome, Mozilla Firefox, Opera Mini, Safari, UC browser etc.

        We develop web service using:

        • .Net (Framework)
        • C# (C sharp) Programming Language
        • IIS Web Server
        • MsSQL database

        We develop our web application using:

        • Django (Framework)
        • Python (Programming Language)
        • Material Design Pattern
        • HTML 5
        • CSS

        We develop 3rd Party Plug In for Moodle or any others System using:

        • PHP programming Language

    5. Design and Implementation Constraints

      This is a distributed service. So, The cloud platform is responsible for receiving the data from any plugins that build for this service and also able to sending data. The system will have a feature to integrate with other third party applications (like Moodle). So the constants will be: For service we are using .Net Framework, we will use Django (a python framework) for cloud platform, for ios we use swift and for android we use Java programming language. And for Moddle we use Php.

    6. Short User Documentation

      User can use the service from two places. They can use it from our website or they can edit their document by signing on Moodle.

      Website: A user needs to sign up to DTS to get the DTS service. After signing up, the user has to upload any documents that he/she wants to tag and then can publish the documents to others.

      Moodle: DTS comes with Moodle plugin. So all DTS users can get the full DTS service in Moodle environment.

    7. Assumptions and Dependencies

      To use our service all a user need to do is add the DTS (Document Tagging System) plugins to which they want to use the service (Moodle), if they use it from our website then they do not need any plugin to tag any documents, after uploading the file for tagging, the user simply can start to tag the documents.

      The system needs a webserver and a database and we are using IIS as webserver as it comes with .Net the backend framework and as database we are using MsSQL.

    This part will consist the four most important interfaces in our system. Each and every part of this section is important for our clients and user. It makes it easier for them to understand part of the system through the following part:

    1. User Interface
    2. Hardware Interface
    3. Software Interface

    Each part's description are given below:

    1. User Interface

      There will be three type of user interfaces. These interfaces required characteristics between our software and our user. For our user interfaces we will use bootstrap material UI in Django and for the backend we will use .NET. Three types of interfaces are:

      1. Moodle Plugin

        For this interface we will use our ULAB Moodle's basic interfaces. This interface plugin will be on PHP. In moodle we use an editor for PDF, it's Moodle editor for annotation. In this editor, there is some awesome features. There will some button with some amazing features. Some features be like:

        1. Add tag:

          There will some default tag for the user's to use. User can add some tag too by their own choice. Just he/she needs to select some text and tag. We will provide some features like tag color, some icon to use as a tag etc.

        2. Comment Tag:

          User can tag some texts or a line or multiple line and also a user can comment on those tags. They can also add many icons line right or wrong sign. And also can tag line by many type of colors.

        3. Add Multiple PDF files:

          A user can add one or multiple PDF files in moodle. One user can also can edit or comment or tag on those files if user have permission.

        4. View all PDF files:

          All users can view PDF files form a group if user is in the group.

      2. Web Application

        Just like the Moodle we will use our own web application for the user. The features will be more or less same for the both interfaces. User can login in our system by becoming our registered user or by using some third party authentication. Options will be provide through our system.

      3. Android Application

        Like web application and iOS application a user also can have our android application in his/her smartphone. It's interface will be same as web application(web view) and iOS application so that user can understand it easily. Also users can upload and annonate PDF files from android application and all the options web application and iOS application have.

      4. iOS Application

        Like web application and android application a user also can have our iOS application in his/her iphone. It's interface will be same as web application(web view) and android application so that user can understand it easily. Also users can upload and annonate PDF files from our iOS application and all the options web application and android application have.

    2. Hardware Interface

      In hardware interface, at first we need to know how many user will use this system and how big the database will. It relies on that the system, which will require a database to store data. If there's some problems occur while the user to tag the document in-case of load shedding or bad Internet connection, our database should have backup capabilities. After that the system will automatically synchronize data from the user. For this interface it requires windows machine for client application, Ubuntu machine for web application and Moodle plug in.

    3. Software Interface

      For software interface, we will use Moodle as a software. Moodle's basic interface will use for this. And we will use Moodle our software and interface. There will be a moodle interface and user will set those rules or PDF. For the view of our web application we will use web browser. The rest will be same as hardware interface.

    4. Communication Interface

      The system is hosted on a server name Microsoft Azure. So, it need HTTP protocol to communicate with the server. We will use our main service API to communicate our applications. Communication between the user and the system can be established through our interfaces. The data transfer rate in our database will be on normal process. We will use browser and for every possible step the asynchronous synchronization process is available for the user. If any problem occurs no data will get lost in our system. For the better communication we will use third party authentication system. User can logged in our system via gmail, face-book, linked-in, twitter and etc. This system has the permission to access in all popular web browsers that can interact with HTML pages. The users can communicate via http/https (port: 80/443) connection and the client will also need to communicate via http/https(port: 80/443) connection.

      Our service is user friendly and reliable. Our main service(.NET) is connected with moodle, android, iOS and also with web. Performance will measure by user's CPU (Central Processing Unit), internet connection speed. But main performance (run time) of our backend service is very sufficient. Our service can take those tags in servers at a real time. Since our service will be connected to every applications(web, moodle, iOS, android), so our service design that way so that it can done its work properly and efficiently.
      A user shouldn't be worry about our service performance. An admin person can create a user or client, in that case our service will take care of performance of admin users or clients. If user work online then data will be saved in real time and save those data in cloud. Our service will be integrated with third party software (with permission) and it's performance will be same reliable.

    2. Safety and Security Requirements

      A user can sign in our system by many ways. It can be our service verification or oAuth or Single sign-on (SSO) for verification or credential. So a user should not be worry about their security. Credentials will be safe and sound in our service.
      Also for authorization our service will take care of authorization of every user. A user should not be worry about our service authorization and can give restrictions on users. Restriction can be view, edit, view and edit etc. So it will be handle as usual.
      At the end, every safety and security is maintain good so user shouldn't worry about authentication and authorization.

    3. Software Quality Attributes

      Software quality attributes specify the required reliability, availability, security, maintainability, robustness, portability of the software system.

      Reliability: The system should able to run continuously a long duration of time and not suffer from system slowdowns or crashes for memory leaks.

      Availability: The information can be send again for verification if the internet service lost during sending information to server.

      Security: Unauthorized access to the system and its data is not allowed. Secure the communication between the system and server.

      Maintainability: The application should be easy to extend, so in order for future functions to be implemented easily to the application. The code should be indicted in a way that it favors implementation of incipient functions.

      Robustness: The system is need to enough robust to successful a transaction and ability of a system to cope with errors during execution.

      Portability: The software should be able to run on any Microsoft windows predicated platform.
